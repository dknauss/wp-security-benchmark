LLM Lies Log
=============
Fabricated, outdated, and unverified claims generated by LLMs and
committed to this repository without verification against live sources.
This file exists for prevarication verification and prevention.

Audited: 2026-03-12 (initial creation — 3 entries from Codex audit)
Repos:   wp-security-benchmark


1. WRONG HEADING LEVEL — Controls verification command
   Files:  docs/current-metrics.md (Architectural Facts table, Structural
           Integrity table, Verification Script)
   Claim:  `grep -cE '^### [0-9]+\.[0-9]+' WordPress-Security-Benchmark.md`
           would count 50 security controls.
   Reality: Controls use H4 headings (`####`), not H3 (`###`). The command
            returned 0 instead of 50. Correct command:
            `grep -cE '^#### [0-9]+\.[0-9]+' WordPress-Security-Benchmark.md`
   Source:  Running the command against the actual document; inspecting
            the Markdown headings (all controls are `#### 1.1`, `#### 2.3`, etc.)
   Found:  Codex audit, 2026-03-12. Severity: High.
   Fixed:  091718c (Claude Code, same day)

2. WRONG STRUCTURAL PATTERN — Audit/Remediation verification commands
   Files:  docs/current-metrics.md (Architectural Facts table, Structural
           Integrity table, Verification Script)
   Claim:  `grep -c '#### Audit' WordPress-Security-Benchmark.md` and
           `grep -c '#### Remediation' WordPress-Security-Benchmark.md`
           would count 50 Audit and 50 Remediation sections.
   Reality: Audit and Remediation sections use bold inline labels
            (`**Audit:**` and `**Remediation:**`), not H4 headings. Both
            commands returned 0 instead of 50. Correct commands:
            `grep -c '\*\*Audit:\*\*' WordPress-Security-Benchmark.md`
            `grep -c '\*\*Remediation:\*\*' WordPress-Security-Benchmark.md`
   Source:  Running the commands against the actual document; inspecting
            the Markdown structure (bold labels, not headings)
   Found:  Codex audit, 2026-03-12. Severity: High.
   Fixed:  091718c (Claude Code, same day)

3. MISCOUNTED CODE FENCES — Raw-attribute fences omitted
   Files:  docs/current-metrics.md (Architectural Facts table, Structural
           Integrity table)
   Claim:  "Code fences (total): 248" with balance check
           "34 + 214 = 248" (tagged openings + bare closings = total).
   Reality: 250 total fences. Two raw-attribute opening fences (```{=html})
            were not counted. The correct balance is:
            34 (tagged) + 2 (raw-attribute) + 214 (bare closing) = 250.
   Source:  `grep -c '^\`\`\`' WordPress-Security-Benchmark.md` returns 250;
            `grep -cE '^\`\`\`\{' WordPress-Security-Benchmark.md` returns 2.
   Found:  Codex CI validator audit, 2026-03-12. Severity: Medium.
   Fixed:  aa9e951 (Codex, same day)
   Notes:  The fence-counting logic assumed all opening fences had a
            language tag (`[a-z]`). Pandoc raw-attribute fences (`{=html}`)
            start with `{`, not a letter, so they fell through to be
            counted as bare fences, breaking the balance check silently.


ROOT CAUSE
----------
All three errors share the same root cause: writing verification commands
from memory or training data without running them against the actual
document. The commands looked plausible but had never been tested.

This is the same failure class documented in wp-sudo's llm-lies-log.md —
confabulated technical details that sound correct but silently fail.
The CI validator added in aa9e951 now runs every verification command
on push and PR, preventing future drift.

PREVENTION
----------
1. Every verification command in current-metrics.md MUST be run and its
   output compared to the stated value before committing.
2. The CI workflow (.github/workflows/validate-metrics.yml) automates
   this check on every push and PR that touches the document or metrics.
3. When adding a new metric, run the command first, then record the result.
